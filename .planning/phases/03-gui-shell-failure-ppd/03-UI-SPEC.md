# Phase 3 — UI Design Contract

> Visual and interaction contract for the GTK4 + libadwaita GUI shell. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

> **Stack note:** AcerControl is a native GNOME application targeting Ubuntu 24.04 (libadwaita 1.5). Web-style design tokens (hex palettes, px scales, font families) are deliberately NOT specified — Adwaita supplies all of those from the user's system theme (light/dark, accent color, fractional scaling). This contract specifies *which Adwaita primitives, style classes, and copy strings* are used. Any executor invention of pixel sizes or hex colors is a contract violation.

## Spacing Scale

Adwaita widget defaults — no custom margins, padding, or grid spacing is declared at the GUI level in Phase 3. `Adw.PreferencesGroup`, `Adw.StatusPage`, `Adw.Banner`, `Adw.HeaderBar`, and `Adw.ToolbarView` provide the spacing contract. Custom `Gtk.Box` containers, where required (e.g. action button row inside a StatusPage child), use Adwaita's documented `spacing=12` default (matches GNOME HIG).

| Token | Value | Usage |
|-------|-------|-------|
| Adwaita default | (theme-supplied) | All widget margins, padding, header bar height, banner padding |
| Box spacing | 12px | The single permitted constant for hand-built `Gtk.Box` rows (matches GNOME HIG) |

Exceptions: none. Any executor-introduced literal pixel value other than `12` (in `Gtk.Box(spacing=12)`) is a contract violation — open a planner ticket instead.

## Color

Adwaita system theme. Light mode, dark mode, and user accent color all follow GNOME settings. **No custom palette ships in Phase 3.** Color semantics are expressed via Adwaita / GTK style classes:

| Role | Style class | Surface in Phase 3 |
|------|-------------|--------------------|
| Warning (in-band, non-blocking) | `.warning` | PPD banner; modprobe-blacklist banner; coretemp-missing banner |
| Error (blocking) | `.error` | Critical-failure StatusPage icons (e.g. `acer_wmi` not loaded, `platform_profile` missing) |
| Suggested (primary remediation) | `.suggested-action` | StatusPage action buttons: "Load module", "Reload with predator_v4=1", "Disable PPD" |
| Destructive | `.destructive-action` | Reserved — NOT used in Phase 3 (no irreversible actions ship this phase; `mask --now` is technically reversible via `unmask`) |
| Monospace | `.monospace` | About → Diagnostics `Gtk.TextView` only |

Accent color usage — reserved for: Adwaita's automatic application of the user's GNOME accent color to focused widgets, `.suggested-action` buttons, and the system-supplied selection highlight. **No manual accent-color CSS is written.**

## Copywriting Contract

Tone: **terse, technical, descriptive** — match `gnome-control-center` and `tlp-rdw`. Never marketing-imperative ("Oh no, something went wrong!" is forbidden). State the failure, state the fix, label the button as `verb + object`. Sentence-case. Toasts end with a period (Adwaita HIG); button labels do not.

### Banner (PPD active — verbatim per ROADMAP success criterion 3)

| Banner button label | `Disable PPD` |
| Banner style class | `.warning` |
| Banner `revealed` default | `true` (visible when `features.probe()` reports PPD active) |

### Banner (acer_wmi blacklist detected — warning)

### Banner (coretemp hwmon missing — warning)

| Element | Copy |

### Toast copy

| Trigger | Copy |
|---------|------|
| User pressed Escape on polkit dialog (any wrapper) | `Authentication cancelled.` |
| Disable PPD success | `power-profiles-daemon disabled.` |

Toast `priority`: `Adw.ToastPriority.NORMAL` for all of the above. Timeout: Adwaita default (5 seconds). All toasts surface via the single `Adw.ToastOverlay` that wraps `Adw.ToolbarView`'s content.

See **StatusPage Copy Table** below for the full per-blocker matrix. Button labels are imperative `verb + object`: `Load module`, `Reload with predator_v4=1`, `Disable PPD`, `Refresh`.

### Empty/placeholder state (no blockers, Phase 3 has no main content yet)

|---------|------|

| Icon | `applications-system-symbolic` |

This placeholder ships only because Phase 3 isolates the GUI shell from the profile-button work (Phase 4). It is removed when Phase 4 lands.

### Destructive actions

None ship in Phase 3. `acercontrol-disable-ppd mask` is reversible via `unmask`; `acercontrol-reload-acer-wmi` is reversible by reloading without args. Therefore: no `Adw.AlertDialog` confirmation prompts, no `.destructive-action` buttons.

## StatusPage Copy Table (per FeatureReport check class)

The GUI's first action on `do_activate` is `features.probe()`. The returned `FeatureReport.checks` are partitioned into **blocker** (full-window StatusPage) and **warning** (banner above main content) per CONTEXT decision #3. If multiple blockers exist, the **first** in declared `checks` order is rendered; user resolves it, presses **Refresh**, the next blocker (if any) is rendered.

### Blockers — full-window `Adw.StatusPage`

| Probe key | Severity | `icon-name` | Title | Description | Action button | Polkit action invoked | Post-action |
|-----------|----------|-------------|-------|-------------|---------------|----------------------|-------------|
| `acer_wmi-not-loaded` | blocker | `dialog-error-symbolic` | `acer_wmi module not loaded` | `The acer_wmi kernel module is required for performance control. Click below to load it with predator_v4=1.` | `Load module` | `org.acercontrol.reload-acer-wmi` | re-probe after wrapper exit; if still failing, show same StatusPage with a toast `Module load failed. Check journalctl.` |
| `predator_v4=N` | blocker | `dialog-warning-symbolic` | `Predator mode disabled` | `acer_wmi is loaded without predator_v4=1. Performance, turbo, and LED features are unavailable until the module is reloaded.` | `Reload with predator_v4=1` | `org.acercontrol.reload-acer-wmi` | re-probe after wrapper exit |
| `platform_profile-missing` | blocker | `dialog-error-symbolic` | `platform_profile interface unavailable` | `/sys/firmware/acpi/platform_profile does not exist. This usually means the kernel is older than 5.12 or ACPI is missing the platform_profile object. No userspace fix is possible.` | `Refresh` (footer-only; no remediation button) | none | re-probe |
| `no-acer-hwmon` | blocker | `dialog-error-symbolic` | `Acer sensors unavailable` | `No hwmon device named "acer" was found. Fan and temperature monitoring will not work. Confirm acer_wmi is loaded with predator_v4=1.` | `Refresh` | none | re-probe |

### Warnings — `Adw.Banner` above main content

| Probe key | Severity | Surface | Style class | Action |
|-----------|----------|---------|-------------|--------|
| `ppd-active` | warning | banner with single button (HeaderBar primary menu hosts the "About PPD" entry) | `.warning` | `Disable PPD` button → `org.acercontrol.disable-ppd` wrapper; "About power-profiles-daemon" menu entry → in-app explainer dialog (always reachable, not gated on banner state) |
| `acer_wmi-blacklisted` | warning | banner, no button | `.warning` | none (read-only — Phase 3 ships no unblacklist remediation) |
| `coretemp-missing` | warning | banner, no button | `.warning` | none (Phase 5 will degrade the CPU package bar to `—`) |

**StatusPage construction contract:**
- All blocker StatusPages are built by factory functions in `acercontrol/gui_status_pages.py`, one per `probe key`.
- The `Refresh` button is a **footer secondary action** on every StatusPage (placed below the primary action button). Implementation: `Adw.StatusPage.set_child` with a `Gtk.Box(orientation=VERTICAL, spacing=12)` containing primary button (when present) then `Refresh` button.
- The `Refresh` button has the `.flat` style class (or no class) — it is NOT `.suggested-action`. The primary remediation button IS `.suggested-action`.
- When `features.probe()` returns `ok=true`, no StatusPage is rendered; the main content area shows the placeholder card from the **Empty/placeholder state** copy above.

**GUI-08 enforcement (forbidden literals in user-facing strings):**
- Forbidden in `acercontrol/gui.py`, `gui_window.py`, `gui_status_pages.py`, `gui_banner.py`: the literals `"low-power"`, `"quiet"`, `"balanced"`, `"balanced-performance"`, `"performance"`. (Note: `"performance"` may legitimately appear as part of `Adw.HeaderBar` window title strings like `"AcerControl"` or in service-name strings like `"acer-performance.service"` — the contract is "kernel profile values never reach user-visible labels," not "the substring 'performance' is banned." A grep gate that searches for these tokens as standalone string literals in `Adw.StatusPage`/`Adw.Banner`/`Gtk.Label` text is the test in 03-VALIDATION.md.)
- Exempt: `acercontrol/gui_about.py` — the Diagnostics carve-out, where raw `features.probe()` JSON renders verbatim.
- All UI labels for profile names go through `acercontrol.profiles.kernel_to_profile()` (Phase 4 surface; Phase 3 does not render profile names anywhere except About → Diagnostics).

## PPD Banner Two-Action Contract (the locked decision)

**Decision: Fallback path — single banner button + "About PPD" entry in `Adw.HeaderBar` primary menu.**

> **Revision 2026-05-16:** Original option (a) (Pango `<a href>` link inside the banner title) is EMPIRICALLY DISCONFIRMED by Phase 3 research. `Adw.Banner` source @ libadwaita 1.5.4 exposes ONLY a `button-clicked` signal; its internal `GtkLabel` is template-private with no public accessor, so `activate-link` cannot propagate. Promoted the fallback that was already documented in this section to the locked path. The verbatim ROADMAP banner copy is preserved by moving "Learn more" off the banner and into the HeaderBar primary menu as "About power-profiles-daemon" — both affordances remain reachable; the banner just stops trying to render a link it can't handle.
>
> Primary source: `adw-banner.c` @ libadwaita 1.5.4 + [class.Banner — libadwaita 1-latest](https://gnome.pages.gitlab.gnome.org/libadwaita/doc/1-latest/class.Banner.html). Full verification chain in `03-RESEARCH.md` Landmine #1.

| Element | Implementation |
|---------|----------------|
| Banner title (`Adw.Banner.set_title()`, plain text — NO Pango link markup) | `power-profiles-daemon is running and will overwrite profile changes` |
| Banner button (`Adw.Banner.set_button_label("Disable PPD")` + `connect("button-clicked", on_disable_ppd)`) | Invokes `acercontrol.privilege.run_privileged()` against the new `acercontrol-disable-ppd` wrapper |
| "Learn more" affordance | NOT on the banner. Instead: `Adw.HeaderBar` primary menu entry labelled "About power-profiles-daemon" opens the existing in-app `Adw.Window` PPD explainer (same dialog spec below). The menu entry is visible at all times (not gated on PPD being active) so users can read the explainer pre-emptively or after dismissing the banner. |
| Dismissal | `Adw.Banner` close button (`set_revealed(False)` on click); in-memory `dismissed` flag held on `MainWindow`; banner hidden for the rest of the session |
| Re-surface contract | `MainWindow.show_ppd_banner(force: bool = False)` — when `force=True`, ignores `dismissed` flag and shows the banner. Phase 4's mismatch handler calls this on revert detection. Cold start: dismissed flag is in-memory only, so a fresh launch re-shows the banner if PPD is still active. |

**ROADMAP success criterion 3 compliance:** the ROADMAP says the banner offers `[Disable PPD]` and `[Learn more]`. With the API constraint, `[Disable PPD]` is the banner button and `[Learn more]` is reachable from the HeaderBar primary menu (one click vs two-on-banner). Both affordances remain a single click from the main window. The banner copy itself is verbatim per the ROADMAP minus the unsupported `[Learn more]` rendering — researchers' option (b) (AlertDialog wrapper) was rejected because it forced two clicks for either action.

**Out of scope for Phase 3:** If a future libadwaita release exposes the internal banner label or adds a second button slot, this decision can be revisited. Not a v1+ concern.

## Transient State Contract (during pkexec call)

When the user clicks **Load module**, **Reload with predator_v4=1**, or **Disable PPD**:

| Phase | UI state |
|-------|----------|
| 0. Click | Button becomes insensitive; label changes to `Authenticating…`; optional `Gtk.Spinner` placed beside the label inside the button (planner picks — Adwaita does not provide a built-in spinner-button widget) |
| 1. polkit dialog open (foreign process) | UI remains insensitive; window does NOT spawn its own modal — the polkit agent is its own dialog |
| 2a. Auth succeeded → wrapper running | Button stays insensitive with `Authenticating…` label (the wrapper completes in milliseconds; no separate "Working…" state needed for Phase 3 wrappers — modprobe takes ≤2s, systemctl mask is sub-second) |
| 2b. Auth cancelled (`PrivilegedResult.cancelled == true`, exit 126) | Button re-enabled; original label restored; `Adw.Toast` `Authentication cancelled.` shown via `ToastOverlay` |
| 3a. Wrapper exit 0 | `features.probe()` re-runs; routing re-evaluates; success toast (e.g. `power-profiles-daemon disabled.`) shown |
| 3b. Wrapper exit non-zero, non-126 | Button re-enabled; original label restored; `Adw.Toast` `Operation failed. See terminal for details.` |

`Refresh` button (footer of every StatusPage) does NOT enter this transient state — it is a pure read-only re-probe and completes synchronously in <50ms. It just calls `features.probe()` and re-renders.

## Interaction Flow

```
launch acercontrol-gui
        │
        ▼
single-instance check (Adw.Application.do_activate)
        │
        ├─ existing instance present → window.present() ; exit
        │
        ▼
construct MainWindow
        │
        ▼
features.probe() runs synchronously
        │
        ├─ FeatureReport.ok == false (any blocker)
        │       │
        │       ▼
        │   render Adw.StatusPage for first_blocking_failure
        │   (warnings banners suppressed in blocker mode)
        │       │
        │       ▼
        │   user clicks remediation button OR Refresh
        │       │
        │       ▼
        │   wrapper invoked via privilege.run_privileged()
        │       │
        │       ├─ cancelled  → toast "Authentication cancelled." ; remain on StatusPage
        │       ├─ failed     → toast failure ; remain on StatusPage
        │       └─ succeeded  → re-probe → loop top
        │
        └─ FeatureReport.ok == true
                │
                ▼
            render placeholder Adw.StatusPage
            ("Profile controls coming in Phase 4")
                │
                ▼
            evaluate warnings → for each warning probe key,
            stack one Adw.Banner above main content
                │
                ▼
            user may click Disable PPD → wrapper → re-probe → banner removes itself
            user may click Learn more  → modal Adw.Window
            user may dismiss banner    → in-memory flag, hidden until cold start or force=True
```

## Accessibility

Phase 3 accessibility contract — keyboard navigation only (screen-reader passes Adwaita default behavior; no custom ARIA-style annotations needed):

| Surface | Default focus | Tab order |
|---------|---------------|-----------|
| Blocker StatusPage with action button | Primary action button (`Load module`, `Reload with predator_v4=1`, or `Disable PPD`) | Action button → Refresh button → header bar primary menu → window controls |
| Blocker StatusPage without action (read-only) | Refresh button | Refresh button → header bar primary menu → window controls |
| Main content (no blockers, banners present) | First banner's button if present, else first banner's close button | Banner button → banner close button → next banner → header bar primary menu |
| Learn-more dialog | Close button | Close button (only interactive widget); `Escape` closes |
| About dialog | Default Adwaita focus order | Adwaita default; `Escape` closes |

All buttons are reachable via Tab. No custom focus traps. Banner close button (visible when `use_revealer=true` and a close button is enabled) is reachable via Tab.
